成果物ごとの役割
=========================

概要
-------------------------

本資料では、チケット管理で扱う成果物の役割と、相互参照の考え方を整理します。
全体の流れは[[チケット管理の流れ]]を参照してください。

それぞれの成果物の役割
-------------------------

### アイデア

#### いつ使う

- ふと思いついたとき
- 「もしかしたら」という仮説があるとき
- 技術的に可能かまだ分からないとき

#### 記録する内容

- どんなアイデアか
- なぜ思いついたか
- どんな価値がありそうか

#### 次のステップ

実現方法や方向性が明確になったら要望に昇格します。
目安は、実現したい内容を一文で説明でき、期待する効果を言語化できている状態です。

### 要望

#### いつ使う

- 「こうなってほしい」が明確になったとき
- やりたいことははっきりしているとき
- 具体的な方法はまだ決まっていないとき

#### 記録する内容

- 何を実現したいか
- なぜ必要か
- 期待される効果

#### 次のステップ

前提条件や実行手順が明確になったらタスクに昇格します。
目安は、作業内容・前提条件・完了条件をそれぞれ箇条書きで記載できる状態です。

### 不具合

#### いつ使う

- システムが正しく動作していないとき
- 仕様通りに実装されていないとき
- 既存機能が壊れたとき

#### 記録する内容

- 不具合の内容
- 再現手順
- 期待される動作と実際の動作
- 影響範囲

#### 種別の変更について

基本的に種別は変更せず、不具合のまま管理します(瑕疵として高優先)。
ただし、調査の結果、以下のような場合は種別を変更します。

- UI/UXが分かりにくく誤解を招く → タスクに変更(改善課題)
- 仕様誤解だが説明不足 → クローズまたはドキュメント改善タスク

### タスク

#### いつ使う

- 着手可能な状態になったとき
- 作業内容が具体的に定義できたとき
- 完了条件が明確になったとき

#### 記録する内容

- 具体的な作業内容
- 前提条件
- 期限
- 完了条件

#### 次のステップ

作業に着手する際、必要に応じて作業プランとチェックリストを生成します。

#### 相互参照

タスクの「関連ファイル」セクションに、作業プランとチェックリストへのリンクを記録します。

### 作業プラン

#### いつ使う

- 複数ステップの作業があるとき
- 技術的判断が必要なとき
- 作業履歴を詳細に記録したいとき

#### 記録する内容

- 作業手順
- 実装方針
- 技術選定
- 判断理由
- 進捗状況

#### 主な用途

- 実装前の設計レビュー
- 作業中の判断記録
- 中断後の状況復元

#### 相互参照

作業プランの「関連ファイル」セクションに、元のチケットとチェックリストへのリンクを記録します。
これにより、作業中でも原点(チケット)と基準(チェックリスト)を確認できます。

### チェックリスト

#### いつ使う

- 品質基準が複雑なとき
- 確認項目が多いとき
- 客観的な合否判定が必要なとき

#### 記録する内容

- 確認項目
- 品質基準
- 合格条件

#### 主な用途

- 実装前の設計レビュー
- 実装中の中間確認
- 実装後の最終確認
- 作業プラン変更時の方向性確認

#### 相互参照

チェックリストは、元のチケットと作業プランへのリンクを持ちます。
特に重要なのは、作業プランが変更された際に、チェックリストで方向性を検証できることです。

設計レビューの重要性
-------------------------

### なぜ設計レビューが必要か

実装してから「これじゃない」と言われることを防ぐためです。

### どうやってレビューするか

作業プランとチェックリストを突き合わせます。

```text
作業プラン: 予定している作業
チェックリスト: 満たすべき要件

→ 作業プランの内容でチェックリストを満たせるか確認
```

### レビューで見つかる問題の例

- 実装漏れ: 「この要件が作業プランに入っていない」
- 品質不足: 「エラーハンドリングの計画がない」
- 過剰実装: 「この作業は要件に入っていない」

チェックリストの羅針盤としての役割
-------------------------

### 作業プラン変更時の方向性確認

作業中に新しいタスクが必要になったり、既存タスクの変更が必要になることがあります。
このとき、チェックリストは「本来の問題解決から逸脱していないか」を検証する基準になります。

### チェックリストと作業プランの関係

```text
チェックリスト: チケット要件に対して不変の基準(羅針盤)
  ↓ 検証
作業プラン: 柔軟に変更可能な実行計画(地図)
```

作業プランは状況に応じて柔軟に変更しますが、チェックリストで定義された「What/Why/How」からは逸脱しないようにします。

### 作業プラン変更時の検証フロー

1. 作業中に新しいタスクが必要と判断します。
2. 作業プランに追加・変更します。
3. チェックリストと突き合わせます。
    - このタスクは「What(作業内容)」に含まれるか
    - 「Why(目的)」の達成に貢献するか
    - 「How(実行方法)」の範囲内か
4. 判定します。
    - 範囲内 → 作業続行
    - 範囲外 → スコープ外として、別チケット化を検討

### よくある逸脱パターン

#### パターン1: 機能追加の誘惑

```text
元のタスク: ログイン機能のバグ修正
作業中: 「ついでにパスワードリセット機能も追加しよう」

→ チェックリストで検証
  What: バグ修正(機能追加ではない)
  判定: スコープ外 → 別タスクとして記録
```

#### パターン2: 技術的興味による拡大

```text
元のタスク: APIレスポンス速度の改善
作業中: 「この機会に全体をマイクロサービス化しよう」

→ チェックリストで検証
  How: レスポンス速度改善(アーキテクチャ刷新ではない)
  判定: スコープ外 → 別タスクとして記録
```

#### パターン3: 問題の連鎖

```text
元のタスク: データ移行ツールの作成
作業中: 「データに不整合があるから、先にデータクレンジングしよう」

→ チェックリストで検証
  前提条件: データ移行が目的(データ品質改善は別問題)
  判定: 関連するが別タスク → 依存関係として記録
```

### 相互参照の重要性

チケット、チェックリスト、作業プランは相互にリンクされており、いつでも原点と基準に立ち返ることができます。

```text
チケット(関連ファイル)
  ├ チェックリスト: 方向性確認用
  └ 作業プラン: 実行計画

作業プラン(関連ファイル)
  ├ チケット: 元となる作業指示書
  └ チェックリスト: 作業の方向性確認用
```

この相互参照により、作業が長期化しても、複数回に分けて進めても、
常に原点(チケット)と基準(チェックリスト)を確認できます。
